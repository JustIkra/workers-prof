Маппинг метрик из .docx → MetricDef

Цель: унифицировать извлечённые из трёх типов отчётов значения в набор MetricDef.{code,name,unit} для расчёта и рекомендаций.

Подход
- Для каждого типа отчёта определить локаторы таблиц (заголовки/позиции) и соответствие заголовков строк/колонок метрикам.
- Значения берём ТОЛЬКО из числовых ярлыков/ячеек (например, «6,4», «7,6» на барах). Символические оценки «++/+/−/--» игнорируются и не конвертируются в числа.
- Нормализовать числовые форматы (запятая/точка) и единицы; допустимый диапазон 1..10 (или 0..100 при процентной шкале — отдельно оговорить).
- Хранить конфигурацию в YAML, загружать на старте воркера.

Скелет YAML конфигурации
reports:
  REPORT_1:
    table_locators:
      - by: header_regex
        pattern: "(?i)ключевые компетенции|итоги"
    header_map:
      # "текст в отчёте" : CODE в MetricDef
      "Ясность речи": COMMUNICATION_CLARITY
      "Работа в команде": TEAMWORK
    value_source: bar_label   # берём число с ярлыка на баре/в ячейке
    number:
      decimal: ","

  REPORT_2:
    table_locators:
      - by: page_index
        index: 2
    header_map:
      "Точность планирования": PLANNING_ACCURACY
    value_source: cell_column
    value_column: 4
    number:
      decimal: ","

  REPORT_3:
    table_locators:
      - by: caption_regex
        pattern: "(?i)оценка компетенций"
    header_map:
      "Клиентоориентированность": CUSTOMER_FOCUS
    value_source: cell_column
    value_column: "Значение"

Правила нормализации
- Триминг и унификация регистра заголовков при сравнении.
- Числа: заменить пробелы, использовать decimal из конфигурации; отбрасывать токены, содержащие «+» или «-».
- Если заголовок не найден в header_map — логировать и отправлять на ручную валидацию.

Процесс внесения маппинга
- Извлечь пилотные таблицы из примеров отчётов.
- Заполнить header_map фактическими названиями строк/колонок.
- Обновить MetricDef (code, name, unit) консистентно с маппингом.
